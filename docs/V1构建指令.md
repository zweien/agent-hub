## 0. 读者约定（给 coding agent 的硬规则）

1. **只做本文档列出的内容。** 报告里出现的 GraphRAG / A2A / Deep Agents / ColQwen2 / 多 agent / 时序图谱等,**V1 一律不实现**(见 §6 砍掉清单)。
2. **三条核心链路必须带最小验证**(§5),这是不可妥协的安全栏,即使拖慢也要做。
3. **技术选型已定**(§2),不要自行替换框架/库。如遇阻塞,记录在 BLOCKERS.md 并停下问人,不要擅自换方案。
4. **保留可回放与可观测**:sandbox 执行、agent 输出、知识入库,都要有日志/产物留存(V1 用最简方式,见 §5)。
5. **模块边界与记忆纪律(必守)**:
   - `agent_runtime/` **不 import 业务层**(api/),不反向依赖 knowledge/tools/sandbox_mgr。
   - **`agent_runtime/` 不持久化任何跨会话记忆**——会话结束记忆清空;要"记住"历史只能调 `knowledge/` 检索。
   - **`knowledge/` 入库永远经人工审核**,agent 不能自动写知识库(防噪声,守决策 #6);agent 可在会话里"建议存档",但入库是人决定的。
   - 前后端经 **OpenAPI 契约**交互,不在前后端间共享纠缠的类型。

## 1. V1 要交付什么（一句话）

> 一个 **BS 架构的 Agent 工作台**:少数 A 类用户能**配置并发布** agent(挂 prompt/工具/知识库/sandbox),多数 B 类用户能**对话使用**已发布的 agent 并查看其配置;先打通**一个垂直优化场景**(参数搜索→沙箱仿真→结果分析)作为端到端验证链路。

### V1 成功标准（可验证）
- [ ] A 类用户能在 Web 上配置一个 agent(system prompt + 工具 + 知识库 + sandbox),发布给团队。
- [ ] B 类用户能用该 agent 对话,agent 能调用 sandbox 跑一段脚本(含 agent 现写的代码)并返回结果。
- [ ] 至少**一个垂直场景端到端跑通**(§4),且有回归用例(§5.2)。
- [ ] sandbox 执行有日志、可回放(§5.1)。
- [ ] 知识库支持人工上传文档 + 向量检索,入库有记录(§5.3)。
- [ ] B 类用户能**只读查看** agent 的配置(prompt/挂了哪些工具)。

## 2. 技术栈（已定，不要替换）

| 层 | 选型 | 说明 |
|----|------|------|
| 后端 | **Python + FastAPI** | 异步,uvicorn |
| Agent 编排 | **LangGraph**(扁平型为主) | V1 只做扁平配置型 agent(LLM + function calling 自主循环),**不做画布编排型**(见 §6) |
| 前端 | **Next.js + React + TypeScript** | UI 用 Ant Design |
| 关系库 | **PostgreSQL** | agent 配置/会话/用户/权限/知识元数据 |
| 向量库 | **pgvector** | 知识库检索(与 PG 一体,零额外组件) |
| 缓存/队列 | **Redis** | 会话缓存 + 异步任务 |
| 模型接入 | **LiteLLM** | OpenAI 兼容协议;接内网 DeepSeek-V4-Flash / MiniMax-M-2.7 |
| 沙箱 | **agent-infra/sandbox** | Docker 部署,**自建薄控制面**(§3) |
| 工具协议 | **MCP + function calling** | V1 只做 MCP 工具接入 |
| 外部能力接入 | **分层：MCP 管动作 + 平台内建管数据** | 见 §2.1；仿真/求解器（动作型）走 MCP 工具；知识库/历史数据（读型）走平台内建检索，不经 MCP |
| 实时通道 | **WebSocket**（必须） | 决策 #15：会话主通道全双工 WS；承载 token 流式、进度推送、§5.4 仿真确认、**§2.3 单人接管** |
| 部署 | **Docker Compose** | V1 单机;生产化(K8s)不在 V1 |

> 说明:LangGraph 在 V1 的角色是"跑一个扁平 agent 循环"的最小内核,**不向用户暴露图编排**。把它当 agent 运行时用,不是当可视化编排引擎用。

#### 2.1 外部能力接入分层（grilling 共识 #11/#12）

agent 要调的外部能力分两类,**走两条不同路径**（不要硬包成一种）:

| 能力类型 | 例子 | 接入方式 | 理由 |
|----------|------|----------|------|
| **动作型**(传参→触发→等结果) | 仿真 API、求解器、触发类服务 | **MCP 工具** | 正是 MCP Tools 用途,带 schema 参数校验 |
| **读型**(检索/查询) | 知识库、历史方案、试验数据 | **平台内建检索层**(pgvector + SQL) | 复用知识沉淀能力(CRAG/审核),包 MCP 反而割裂 |

> 真实仿真软件(商业/自研)均**通过 API 暴露**——它们有自己的运行环境,**不在 sandbox 里跑**。sandbox 只负责 agent 现写的轻量分析脚本(A 类),真实仿真通过 MCP 工具调用其 API。

#### 2.2 仓库结构与模块边界（grilling 共识 #14：模块化单体）

**形态:模块化单体(monorepo + 强制分层包),单(或少数)进程部署,内部边界清晰。** V1 抢速度但不允许搅成一坨——边界是 coding agent 必须遵守的硬规则(见 §0)。

```
agent-hub/
├── backend/                 # FastAPI 单进程,内部分层包
│   ├── agent_runtime/       # ★ 最硬边界:LangGraph agent 内核 + 模式化防护(§5.4)+ MCP 工具调用
│   ├── sandbox_mgr/         # 沙箱控制面(SandboxManager 接口,Docker 实现,§3)
│   ├── knowledge/           # 平台内建检索(pgvector + 上传/复核,§5.3)
│   ├── tools/               # MCP 工具适配(接真实仿真 API,§2.1)
│   ├── api/                 # FastAPI 路由(配置面/对话面/权限),编排上述四模块
│   └── models/              # PG 数据模型(agent 配置/会话/用户/权限/知识元数据)
├── frontend/                # Next.js + React + TS,按 OpenAPI 契约调后端
└── docker-compose.yml       # PG + pgvector + Redis + LiteLLM + sandbox + 前后端
```

**边界纪律(coding agent 必须遵守,违反即 bug):**
- `agent_runtime/` **不 import 业务层**(api/);它只通过明确接口对外("运行一个 agent 会话")。
- `knowledge/`、`tools/`、`sandbox_mgr/` 经接口**被** `agent_runtime` 调用,**不反向依赖** agent_runtime。
- 前后端经 **OpenAPI 契约**交互(FastAPI 自动产出,前端按契约生成 client),不在前后端间共享纠缠的类型。
- 各模块演化到 V2(多 agent、编排型、GraphRAG)时,**增量替换而非推倒**——这正是模块边界的价值。

#### 2.3 会话交互与人机接管（grilling 共识 #15/#15.1/#15.2）

WebSocket 是 V1 **必须能力**,定位为会话主通道。这源于产品核心价值:**人机协同**——用户能中途接管 agent、暂停/恢复,而非 agent 黑盒跑完给结果。

**WebSocket 事件协议(V1 轻档,协议可扩展、实现只做轻档):**
- **下行(后端→前端)**,带 `type` 字段:
  - `token` — LLM 流式 token。
  - `progress` — 进度通知(检索知识库/调仿真/沙箱执行中)。
  - `action_required` — 需用户确认的动作(§5.4 仿真前置确认),带 `action_id`。
  - `takeover_ready` — sandbox 工作环境(VSCode/终端)已就绪,可接管。
  - `state` — 会话状态变更通知(见状态机)。
- **上行(前端→后端)**:
  - 发消息、确认/拒绝 `action_id`(可携带修改后的参数)。
  - `takeover_begin` / `takeover_end` — 请求接管 / 交还。

**会话状态机(含接管态):**
`idle → running →(遇 action_required)→ awaiting_user →(确认)→ running → … → done`
`running →(takeover_begin)→ human_takeover(agent 挂起)→(takeover_end)→ running`
`*  → interrupted`(§5.4 超时/熔断)

**人机接管(C1,V1 做):**
- 用户点"接管" → agent 挂起 → **sandbox 的 VSCode Server / 终端 / 浏览器对用户开放**(这正是 agent-infra/sandbox 的核心卖点)→ 用户手动操作(改参数、跑脚本、看结果)→ 点"交还" → agent 读最新工作环境状态继续。
- **单人独占**:同一会话同一时刻只有一个操作者(人或 agent),不处理多人并发。
- 接管期间的所有操作(人的、agent 的)**都进可回放日志(§5.1 / C2)**。

**V1 明确不做(推 V2):**
- ❌ 多人实时协同(C3):多客户端同时看/操作同一会话(实时多光标/状态广播/冲突处理)——实时协作引擎级复杂度,与快速原型冲突。
- ❌ 多 agent 实时状态树(WS B 档):V1 不做多 agent,无需任务树状态广播。

> ⚠️ **真实张力(已确认接受):** WebSocket + 单人接管(C1)相比纯 SSE 增加了状态机、连接管理、断线重连、接管语义的工程量,与"快速原型"存在张力。这是为"人机协同"核心价值主动支付的代价。coding agent 实现时须把接管状态机做扎实(实时系统的并发 bug 难调),建议优先写状态机的单元测试。

#### 2.4 会话执行模型:独立于连接（grilling 共识 #19）

会话生命周期**不绑前端连接**——这是 #15(WebSocket)/#16(事件流)/§2.3(接管)三者一致性的必然要求:仿真可能跑几十分钟、接管后用户可能关浏览器离开,会话若绑死连接则全部失效。

**模型(V1):**
- **agent 会话 = 一个后端异步任务**(asyncio task,**V1 不上 Celery**,单进程内即可),独立于任何 WebSocket 连接运行。
- **状态机 + 全部活动持久在事件流(§2.5)**:会话的"真实状态"在事件流里,**不依赖内存中的 agent 对象**。
- **WebSocket 是观察 + 交互层**:前端连上→订阅该会话事件实时推送(token/进度/确认/接管);前端断了→agent 继续跑、事件继续落库;前端重连→读事件流重建当前状态、继续推送。
- **断线重连**:前端重连带 session_id,后端从事件流重放"当前状态 + 最近 N 条事件",前端恢复界面。长任务期间用户可放心刷新/关浏览器。

**V1 不做(留 V2):** 分布式 worker、多节点会话迁移、会话数天级长期挂起。

**对架构的含义(关键):** `agent_runtime/` 的"运行一个 agent 会话"接口(§2.2)真正含义是——**返回/管理一个事件流驱动的、可重连的、脱离前端的异步执行单元**,而非"在请求线程里跑完返回"。这是 §2.2 模块边界的应有之义,现在点破。

#### 2.5 会话数据模型:事件流（grilling 共识 #16）

会话采用**事件流(event sourcing)模型**——这是 §5.1 可回放、§5.4 防护追溯、§2.3 接管记录、§7 协作发布的**共同地基**,四件事共用一套存储,而非各搞一套。

**主存:不可变事件流(唯一事实来源)**
- 每条事件:`{event_id, session_id, timestamp, type, payload, actor(user/agent/system)}`
- append-only、不可变,按 session_id 聚合。
- **事件类型枚举**(coding agent 实现时先定义完整枚举):
  - `message_in`(用户消息)/ `message_out`(agent 回复,可由 token 聚合)
  - `llm_call`(模型/参数/返回/耗时)
  - `tool_call`(工具名/传参/返回,含仿真 MCP 调用)
  - `sandbox_exec`(命令/stdout/stderr/退出码/产出文件)
  - `state_change`(running/awaiting_user/human_takeover/interrupted/done)
  - `action_required` / `action_resolved`(§5.4 确认请求与用户决定)
  - `takeover_begin` / `takeover_op` / `takeover_end`(接管期间人的操作)
  - `interrupted`(熔断/超时,含原因)

**消息流是事件流的投影**:对话界面展示的"用户消息 + agent 回复"由 `message_in`/`message_out` 事件投影而来,**不单独存**,避免双写不一致。

**由此统一支撑:**
- **§5.1 可回放**:按 session_id 取全部事件按序重放,重建任意时刻状态、看到每次工具/仿真/接管细节。
- **§5.4 防护追溯**:熔断/确认事件天然在流里。
- **§2.3 接管记录**:人的 `takeover_op` 进流,事后可追责。
- **§7 协作发布**:一个会话"发布到团队"= 发布其事件流(+ 关键产物文件),成员能看完整过程,不只是结论。

> **成本**:事件流存储量大,但 V1 单团队规模 PG 完全扛得住;append-only 写入简单。复杂度主要在事件 schema 设计——让 coding agent 优先定义完整事件类型枚举与校验。

## 3. Sandbox 控制面（自建，V1 最小版）

agent-infra/sandbox 只是数据面(运行时),**没有开源 manager**(见报告第 6 章)。V1 自建一个**最薄的控制面**:

### V1 控制面职责（最小集）
1. **会话→容器映射**:一个 agent 会话对应一个(或复用)sandbox 容器,记录映射到 PG。
2. **生命周期**:按需 `docker run` 创建、空闲/会话结束 `docker rm` 回收。
3. **GPU 透传**:用 Docker `--gpus`(或 `device_requests`)把 GPU 透传给容器。✅ POC 已验证控制面逻辑就绪(本机无卡待补,见 §9)。
4. **工作环境对人与 agent 共享(支持接管 §2.3)**:sandbox 的 VSCode Server / 终端 / 浏览器既给 agent 用、也能在接管时暴露给用户。✅ POC 已验证 code-server/VNC 全部可用。这是"人机共享工作台"的落点,也是选 agent-infra/sandbox 的核心理由。
5. **日志/产物收集**:容器内执行的命令、stdout/stderr、产出文件,**含接管期间人的操作**,留存并可回放(§5.1)。
6. **基础鉴权**:容器 API 用 API Key。

### V1 实现（✅ POC 后架构优化）
- **生产推荐:经 sandbox HTTP API 调用**(POC 发现)。真 sandbox 暴露 `/v1/shell/exec`、`/v1/file/read|write` 等标准 HTTP API,agent/控制面**经 HTTP 调用 sandbox,无需 docker socket**——更安全、更解耦、更适合多实例。
- 统一抽象 `SandboxManager` 接口(`acquire/exec/release`),**两个后端**:
  - `HttpDialect`(生产推荐):生命周期仍用 docker SDK 起停容器,但**执行/文件操作走 sandbox HTTP API**。POC 已验证此路径可行。
  - `DockerDialect`(POC 已实现,兼容/单机):全程 docker-py exec。
  - 未来 `K8sDialect`(生产高可用):接口不变。
- POC 产物 `backend/poc/sandbox/manager.py` 已实现 DockerDialect;V1 主体补 HttpDialect。

### ❌ V1 不做
- 实例池预热、复杂配额调度、多节点、快照/恢复、gVisor 强隔离。隔离用默认 Docker 容器即可(V1 单团队内网,信任边界可接受;强隔离留待生产化)。

## 4. 端到端验证场景（V1 必须跑通的一个垂直任务）

> 用一个真实场景把"配置→对话→调工具(MCP 气动工具 + 平台检索)→跑沙箱→给建议"整条链路打通。

**场景:机翼气动(升阻比)优化辅助**(气动 POC,grilling 共识 #A1/#A2)
- A 类用户配置一个 agent,挂载:
  - system prompt:你是机翼气动优化助手,根据用户目标(如最大化巡航升阻比)建议机翼参数(展弦比、面积、扭转等)并调用气动工具验证。
  - **动作型工具(MCP):一个"跑气动分析"工具 `run_aero(geo_params) → {CL, CD, L_D, spanwise_CL, ...}`**。V1 接团队真实气动工具;**POC 阶段用轻量替身**(见 §4.1)。
  - **读型检索(平台内建):知识库检索**(历史翼型/机翼方案文档,走 pgvector)。
- B 类用户对话:"我想最大化巡航升阻比,当前展弦比 8、面积 0.5㎡,帮我优化。"
- agent 行为:**检索历史方案(平台内建) → 建议参数方向 → 调气动工具(MCP,传参) → 在 sandbox 里跑自己现写的参数扫描/后处理脚本 → 给出带数据支撑的气动建议**(如"提高展弦比到 12 降低诱导阻力,升阻比提升 X%")。
- 此场景同时验证:**动作走 MCP(气动工具)、数据走平台内建检索、agent 现写代码在 sandbox 跑**(决策 #4/#12)。

### 4.1 气动 POC 替身模型（grilling 共识 #A2：轻量替身优先）

POC 的"气动仿真"用**纯 Python 轻量替身**(升力线/涡格法),不接真 CFD/VSPAero/XFOIL——避免装环境卡死 POC,聚焦验证平台链路本身。

| 实现 | 说明 | POC 用途 |
|------|------|---------|
| **AeroSandbox**(MIT) ⭐优先 | 纯 Python,内置 LLT/VLM/气动优化,现代化 API | 装得上就用 |
| **自写 LLT/VLM ~150 行**(零依赖) | 降级方案 | AeroSandbox 内网装不上时回落 |

**降级策略(POC 特有):** 内网 pip 可能受限 → 优先试 AeroSandbox;装不上回落到零依赖的自写 LLT(纯 numpy)。两者**输出相同 schema**(`run_aero(geo_params) → aero_results`),agent 侧无感知。

**替身输出有气动学理的量**(让 POC 有说服力,非玩具公式):
- 展向升力分布、诱导阻力、Oswald 效率因子、升阻比 L/D。
- 参数扫描(展弦比/面积/扭转)能得出**符合气动直觉的优化结论**(如大展弦比降诱导阻力)。

**真工具接入(POC 之后,V1 正式):** VSPAero/XFOIL/团队自研气动代码,只是**替换 `run_aero` MCP 工具的实现,平台链路骨架不动**(§2.1 分层架构的红利)。

> ⚠️ POC 不验证气动保真度,只验证**平台链路**(配置→对话→调工具→跑沙箱→给建议)能跑通。替身的气动学理只是为了演示时不空洞。

## 5. 三条核心验证链路（不可妥协的安全栏）

### 5.1 会话全链路可回放（基于事件流 §2.5）
- 每次 sandbox 执行、工具调用、LLM 调用、状态转换、接管操作,均作为事件落入**事件流(§2.5)**,不单独起炉灶。
- **可按 session_id 重放查看**整条链路(agent 的每一步、人的接管操作),目的:agent 给的设计建议依据可追溯,防止"跑错了还当对的",也支撑 §7 发布会话给团队看全过程。

### 5.2 Agent 输出回归用例
- 维护 ≥2 个**已知正确答案的回归用例**(如某个历史优化,正确趋势是 X)。
- 跑 agent,验证其建议与已知答案**方向一致**(不必精确相等,LLM 有随机性,检趋势/范围)。
- 这些用例同时是**模型准入评测**的种子(决策 #5 的延伸)。

### 5.3 知识入库人工复核
- 知识库支持上传文档 → 自动切块+向量化入 pgvector,**但入库记录可见、可抽样人工复核**。
- V1 不做复杂审核流,但要有:**上传记录、可标记"已复核/存疑"、可删除**。

### 5.4 运行时安全防护（模式化，grilling 共识 #13）

agent 自主循环 + LLM + 真实仿真 + agent 写代码 = 必然会"跑飞"(死循环/超时/烧算力/乱调仿真)。V1 用**模式驱动的策略层**防护,而非一刀切硬规则。

**三层防护(从外到内,均不可省):**

| 防护层 | 机制 | 说明 |
|--------|------|------|
| ① 硬熔断 | 全局上限:最大轮数 / 最大工具调用次数 / 单工具超时 / 总执行超时 / 单会话 token 上限 | 超限熔断,返回结构化结果。**下限不可完全关闭**(即便 YOLO 也留总执行超时防死循环) |
| ② 真实仿真前置确认 | 动作型 MCP 工具(真实仿真/求解器,有外部副作用)调用前,拦截并弹给用户确认参数 | V1 做"确认/取消"弹窗,不做复杂审批流 |
| ③ sandbox 硬限制 | 容器级 CPU/内存/超时(cgroups),agent 现写代码跑飞只死容器 | 见 §3 |

**三种模式(由 agent 创建者 A 类配置默认,B 类可在上限内切换):**

| 模式 | 仿真前置确认 | 熔断上限 | sandbox 限制 | 适用 |
|------|------------|---------|------------|------|
| **严谨**(默认) | 强制每次确认 | 紧 | 严 | 新场景/生产/不熟悉任务 |
| **标准** | 仅首次或可疑参数确认 | 中 | 中 | 日常迭代优化 |
| **YOLO** | 关闭,agent 自主调 | 宽(但留总超时底线) | 宽 | 可信/可重置沙箱、快速探索 |

**安全不变量(任何模式都不可破):**
- 硬熔断的下限不关;**可回放日志(§5.1)始终开**——YOLO 模式下尤其重要,出事能回看。
- 熔断/确认/拒绝事件都进可回放日志(哪轮、为何被拦、用户批准/拒绝了什么)。
- B 类可切换模式,但**不能突破 A 类为该 agent 设的上限**。

### 5.5 失败处理:暂停 + 解释 + 可恢复（grilling 共识 #20）

场景里"失败"是常态(仿真超时、参数不收敛、API 抖动、LLM 异常)。失败不是终点,是"轮到人决策怎么办"的节点——与 #15 人机协同理念一致。

**失败处理契约(V1):**
- 任何失败(工具异常/熔断/LLM 错误/超时)→ 会话进 `interrupted`,**不结束**,事件流记 `interrupted` 事件(含错误类型 + 上下文 + 可恢复性判断)。
- agent 向用户给**结构化说明**:发生了什么、在哪一步、影响什么,**而非裸报错**。
- 用户的选择:
  1. **重试该步**(可改参数);
  2. **接管**(§2.3 C1)手动处理;
  3. **跳过该步继续**(agent 尝试不带该结果往下走);
  4. **结束会话**(主动放弃,事件流保留可回放)。
- **不可恢复的失败**(鉴权彻底失效、资源不可达)→ 明确告知、进 `failed`、提示需人工介入。

**核心洞察(复用,不重复造):** §5.4 的"仿真确认"、§2.3 的"接管"、本节的"失败处理",本质是**同一种"会话暂停态 + 用户决策 + 继续"模式**——只是触发原因不同。V1 用**一套机制**实现三种触发,复用事件流(§2.5)和会话状态机(§2.3/§2.4)。这正是 #16 事件流 + #19 独立会话带来的红利。
