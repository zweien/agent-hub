# Agent Hub · V2 构建指令（给 coding agent）

> 本文档是 **V2 阶段的唯一执行权威**。配套的《可行性与技术选型报告.md》是全景调研与留档;V1 主体已交付(见《V1构建指令.md》§10)。凡本文档与报告冲突,以本文档为准。
>
> 本文档源自一次 grill-with-docs 访谈,凝结了 **10 个决策**(见末尾共识表)。严格按本文档实现,**不要扩张范围**。标注「❌ V2 不做」的项即使报告推荐,也不实现。

## 决策依赖图（10 个决策的因果脉络，先看这张图）

V2 在 V1 已交付的单 flat agent + 事件流 + 沙箱 + 防护地基上,加两条主轴:**知识(记忆)** 与 **编排(多 agent)**。

```
【V2 双主轴】
 #1 主轴=知识+编排 ──┬─→ #3 知识层=第1层+Wiki骨架(GraphRAG 推 V3)
                    │        ↑
                    │   #2 embedding 源=运维配 Gateway channel(外部阻塞,ADR-0001)
                    │   #4 文档格式=文本+PDF+Office
                    │
                    └─→ #5 编排=子代理委派+画布编排
                            ↑
                         #6 画布与扁平并存(type=flat|canvas)
                         #7 画布编译为 LangGraph StateGraph(ADR-0002)
                         #8 画布 8 种节点(LLM/工具/子代理/条件/入口出口/循环/并行/HITL)
                         #10 子代理类型=A 类预定义

【交付形态】
 #9 一次性全交付(内部实施序:知识→子代理→画布,依赖序)
```

**读图要点:**
- **#2 是外部阻塞**:embedding channel 不配,知识库不可发布。尽早推动运维。
- **#7 是架构锁**:画布 JSON→LangGraph 编译器是 V2 最硬的新工程,决定 #8 节点能力上限。
- **#10 子代理类型为画布的子代理节点提供"类型库"**:flat agent 的 task 工具与 canvas 的子代理节点共用同一套子代理类型定义。
- **知识检索是画布的工具节点可调能力**:V2a 知识库就绪后,canvas 的工具节点能挂"检索知识库"。

## 0. 读者约定（给 coding agent 的硬规则,继承 V1 §0）

1. **只做本文档列出的内容。** 报告里的 GraphRAG / ColQwen2 / A2A / Mem0 / K8s 等(见 §6),**V2 一律不实现**。
2. **继承 V1 的模块边界纪律**:`agent_runtime/` 不 import 业务层;`knowledge/` 被 agent 调用,不反向依赖;入库永远经人工审核(决策 #6 沿用)。
3. **画布编译器复用 V1 执行底座**:canvas agent 与 flat agent 共享 `astream_agent` 流式 / MemorySaver checkpointer / 事件流投影 / 防护栏。不为 canvas 另起一套执行/持久化。
4. **三条核心验证链路沿用 V1 §5**:可回放 / 回归用例 / 知识入库复核。V2 新增画布编译校验链路(§5.4)。
5. **遇阻塞记录 BLOCKERS.md 并停下问人**,不擅自换方案(尤其 embedding channel 未就绪时)。

## 1. V2 要交付什么（一句话）

> 在 V1 单 flat agent 平台上,加 **知识库(向量 RAG + 结构化 Wiki 骨架)** 和 **多 agent 编排(子代理委派 + 画布编排)**,使 agent 能检索团队历史知识、委派子任务、并支持 A 类用画布可视化编排多节点工作流。

### V2 成功标准（可验证）
- [ ] A 类能上传文档(txt/md/pdf/docx/xlsx)→ 自动切块+embedding 入 pgvector → 入库记录可见、可标记"已复核/存疑"、可删除(决策 #2/#3/#4)。
- [ ] flat agent 能调"检索知识库"工具,RAG 结果进对话上下文(决策 #3)。
- [ ] flat agent 能调 `task` 工具 spawn 后台子代理(按 A 类预定义的子代理类型,决策 #10),子代理并行执行,结果回汇。
- [ ] A 类能在画布(ReactFlow)上拖拽 8 种节点连线成图,保存为 canvas 类型 Agent Config(决策 #6/#8)。
- [ ] canvas agent 运行时把图定义编译为 LangGraph StateGraph 执行(ADR-0002),流式/事件流/防护与 flat agent 一致。
- [ ] 画布图定义非法(环无出口/引用不存在的工具)时,编译阶段明确报错,会话不启动(§5.4)。
- [ ] 现有 flat agent(气动/CAD)不受影响,type 默认 flat。

## 2. 技术栈增量（在 V1 基础上）

| 层 | V2 增量 | 说明 |
|----|---------|------|
| Embedding | **Gateway /embeddings**(运维配 channel,ADR-0001) | OpenAI 兼容,与 chat model 同一网关抽象;**不本地起 sentence-transformers** |
| 文档解析 | **PyMuPDF(pdf) + python-docx + openpyxl** | 文本/md 直读;PDF 提取文字;Word/Excel 解析(决策 #4) |
| 向量检索 | **pgvector**(已在 compose)+ 手写检索层 | V2 不引 LlamaIndex 全家桶;切块+embedding+检索用 ~200 行自写(复用 V1 极简风格) |
| 子代理 | **deepagents SubAgentMiddleware**(已装 0.6.12) | flat agent 增量启用;canvas 子代理节点共用类型定义 |
| 画布前端 | **ReactFlow(xyflow)** | 节点编辑器;存图定义为 JSON |
| 画布后端 | **图定义→LangGraph StateGraph 编译器**(自写,ADR-0002) | 复用 V1 astream_agent/MemorySaver/事件流 |

> **不引 LlamaIndex/LightRAG 全家桶**:V2 知识库只做第1层向量 RAG + Wiki 骨架,切块/embedding/检索逻辑足够简单,自写可控且轻;框架锁定的代价不值得(决策 #3)。

## 3. 知识库（第1层向量 RAG + 结构化 Wiki 骨架）

### 3.1 embedding 源（外部阻塞,ADR-0001）
- **Gateway 当前无 embedding channel**(实测 `/embeddings` 对 bge-m3/bge-large-zh/text-embedding-3-small/Qwen3-Embedding 全部 model_not_found)。
- **V2 阻塞项**:推动运维在 Gateway(`192.168.2.220:3000`)配一个 embedding channel(推荐 `bge-m3`,1024 维,中文好)。
- **平台侧**:走 OpenAI 兼容 `POST /embeddings {model, input}`,与 chat model 同一 `LLM_BASE_URL`/`LLM_API_KEY`。embedding 模型名加进 `models` 表(type 区分 chat/embedding,或单独 config 字段——实现时定,记 ADR 若有锁)。
- **channel 未就绪时**:知识库上传接口返回 503"embedding 源未配置",不静默失败。

### 3.2 文档摄入（异步,继承 V1 asyncio 模式）
- **解析**:txt/md 直读;PDF 用 PyMuPDF 提取文字(决策 #4);docx 用 python-docx;xlsx 用 openpyxl(按行/表切块)。
- **切块**:RecursiveCharacterTextSplitter 风格(中文按段落+长度,chunk_size=500, overlap=50)。不引 langchain 的 splitter,自写 ~30 行。
- **embedding+入库**:`asyncio.create_task` 后台执行(V1 §2.4 模式,不上 Celery)。上传接口立即返回文档记录(status=processing);后台任务切块→批量 embedding→写 pgvector→status=ready;失败 status=failed+error。
- **幂等**:同文档(sha256)重复上传不重复入库。

### 3.3 检索（agent 工具）
- 暴露为平台内建工具 `search_knowledge(query, top_k=5)`(V1 §2.1 读型走平台内建,不经 MCP)。
- pgvector 余弦相似度检索,返回 top_k 片段(含来源文档名+chunk 偏移)。
- flat agent 配置时勾选此工具即可用;canvas 的工具节点也能挂。

### 3.4 结构化 Wiki 骨架（第3层骨架,V2 不建完整治理流）
- **Wiki 表**(PG):`wiki_entries(id, title, content, category, source_session_id, status[pending/approved/rejected], owner_id, created_at)`。
- **V2 只做骨架**:agent 在会话里可"建议存档"(写一条 status=pending 的 wiki_entry,关联 source_session_id);**入库永远经人工审核**(V1 决策 #6 沿用)——A 类在知识库管理页 approve/reject。
- **V2 不做**:版本/溯源/增量更新/GraphRAG(推 V3)。

### 3.5 前端
- 知识库管理页:文档列表(状态徽章 processing/ready/failed)+ 上传 + 删除 + 标记已复核。
- Wiki 骨架页:pending 条目列表 + approve/reject(最简)。

## 4. 子代理委派（flat agent 增量）

### 4.1 启用 SubAgentMiddleware
- `build_agent`(flat 类型)在构造 `create_deep_agent` 时,挂 `SubAgentMiddleware`(已装),参数为该 Agent Config 的子代理类型定义(决策 #10)。
- 主 agent 调 `task` 工具时,按 `subagent_type` 名查类型定义,spawn 后台子代理(deepagents 原生非阻塞并行)。
- 前端 `SubagentCard` 已就绪(V1 已处理 `is_subagent` 事件),无需改。

### 4.2 子代理类型定义（A 类预定义,决策 #10）
- Agent Config 加 `subagent_types` JSONB 字段:`[{name, description, system_prompt, tools[], model}]`。
- A 类在 agent 配置页定义命名子类型(如 `researcher`=检索知识库+总结,`analyst`=沙箱跑分析)。
- **主 agent 不能临时写子代理 prompt**(可控边界,与 V1 可配置理念一致)。

### 4.3 防护栏沿用
- 子代理继承父 agent 的 guard_mode / 熔断上限(V1 §5.4)。
- 子代理的工具调用/沙箱执行进同一事件流(可回放)。

## 5. 画布编排（Canvas Orchestration）

### 5.1 Agent Type 字段（并存,决策 #6）
- `agent_configs` 加 `type` 列(String,默认 `flat`,可选 `canvas`)。
- 现有气动/CAD agent 保持 flat,不迁移。
- A 类建 agent 时选 flat 或 canvas。

### 5.2 图定义数据模型（canvas 类型专用）
- `agent_configs` 加 `canvas_def` JSONB 字段(仅 type=canvas 用):
  ```json
  {
    "nodes": [{"id", "type", "position", "data": {节点参数}}],
    "edges": [{"id", "source", "target", "source_handle", "data": {条件表达式}}],
    "entry_node_id": "..."
  }
  ```
- **节点 type 枚举(8 种,决策 #8)**:
  | type | 用途 | LangGraph 映射 |
  |------|------|---------------|
  | `llm` | LLM 调用(带 system_prompt+model) | node→AIMessage |
  | `tool` | 调工具(内置/MCP/沙箱/检索知识库) | node→ToolMessage |
  | `subagent` | spawn 子代理(引用 subagent_types 名) | node→调 task 工具 |
  | `condition` | 条件分支(基于 state 求值表达式) | conditional edge |
  | `entry`/`exit` | 图边界(用户输入→图→最终答案) | START/END |
  | `loop` | 循环(带退出条件表达式) | 循环边 |
  | `parallel` | 并行 fan-out + join | Send fan-out + 合并 node |
  | `hitl` | 人工审核节点(暂停等用户) | interrupt() |
- **节点参数 data**:每种 type 有自己的 schema(llm 的 prompt/model,tool 的 tool_name,args 映射等)。编译器按 type 分派。

### 5.3 编译器（JSON→LangGraph StateGraph,ADR-0002）
- 新增 `agent_runtime/canvas_compiler.py`:读 `canvas_def` → 建 `StateGraph` → 节点转 callable → 边转 edge/conditional_edge → 编译。
- **编译校验(§5.4 安全栏)**:入口存在、无悬空边、condition/loop 的出口可达、引用的 tool/subagent_type 存在。非法 → 抛 `CanvasCompileError`(带节点 id + 原因),会话不启动。
- **复用 V1 执行底座**:编译出的 graph 经 `astream_agent` 同一套流式/checkpointer/事件流/防护。`astream_agent` 不分 flat/canvas——canvas agent 就是"build_agent 返回一个编译好的 graph 而非 create_deep_agent"。

### 5.4 画布校验链路（V2 新增安全栏）
- 保存 canvas Agent Config 时,前端先做基础校验(无入口/有出口未连)。
- **后端启动会话时强制编译校验**:编译失败 → 返回结构化错误给前端(A 类能定位哪个节点/边错),不启动会话。
- 编译错误进事件流(可追溯 A 类配了什么坏图)。

### 5.5 前端画布编辑器（ReactFlow）
- 新增 `/agents/[id]/canvas` 页(仅 type=canvas 的 Agent Config 进入)。
- 左侧节点面板(8 种拖入)+ 中央画布 + 右侧节点参数面板。
- 保存:序列化为 `canvas_def` JSON,PUT `/agents/{id}`。
- **不做**:节点动画、 minimap(可选小地图)、撤销重做(V2 能用即止)。

## 6. ❌ V2 明确不做（推 V3+）

| 砍掉项 | 原因 |
|--------|------|
| **GraphRAG(LightRAG/MS GraphRAG)** | 案例数不足,价值未到;决策 #3 推 V3 |
| **ColQwen2 多模态图纸检索** | 需 GPU + 多向量索引兼容性未验;图纸检索成痛点再上 |
| **A2A 协议** | V2 纯内部 LangGraph 编排,无跨边界;报告 §4.5.1 中期推荐 |
| **Mem0 / Letta 复杂记忆** | V2 用 RAG + 对话历史 |
| **结构化 Wiki 完整治理流(版本/溯源/增量)** | V2 只骨架(决策 #3) |
| **K8s 生产化 + GPU 多租户配额** | 单团队内网未到量级;报告阶段 3 |
| **多人实时协同(C3)** | V1 已推 V2;V2 仍只单人接管 |
| **LlamaIndex 全家桶** | V2 自写检索层,框架锁不值得(§2) |
| **画布节点动画/撤销重做** | 能用即止(V1 §8 理念沿用) |

## 7. 实施顺序（建议,依赖序;一次性交付但内部按此序推）

> V2 内部分三段,按依赖序:知识 → 子代理 → 画布。每段做完跑验证再进下一段。

1. **知识库地基**(§3.1-3.3):推动运维配 embedding channel(阻塞项,尽早);文档上传+解析+切块+embedding+pgvector;`search_knowledge` 工具。**验证**:上传一篇 PDF→ready→agent 检索命中。
2. **Wiki 骨架**(§3.4-3.5):wiki_entries 表 + agent 建议存档 + A 类审核页。**验证**:会话里建议存档→A 类 approve→可见。
3. **子代理委派**(§4):Agent Config 加 subagent_types;build_agent 挂 SubAgentMiddleware。**验证**:flat agent 调 task→子代理并行→结果回汇→事件流可回放。
4. **画布数据模型 + 编译器**(§5.1-5.3):type/canvas_def 字段;canvas_compiler.py;编译校验。**验证**:手写一个 canvas_def JSON→编译为 StateGraph→跑通简单 2 节点图。
5. **画布前端**(§5.5):ReactFlow 编辑器 + 8 节点 + 保存。**验证**:A 类拖图→保存→B 类运行→流式与 flat 一致。
6. **端到端 + 回归**(§5.4 + V1 §5):画布校验链路;现有 flat agent 回归不破。**验证**:气动/CAD flat agent 仍跑通。

---

## 8. V2 验证链路（继承 V1 §5 + 新增）

- **§5.1 可回放**:沿用——子代理委派/画布节点执行/知识检索全进事件流。
- **§5.2 回归**:V1 气动回归用例不破(flat agent 不受影响)。
- **§5.3 知识入库复核**:沿用——上传记录可见、可标记、可删除;wiki 入库经审核。
- **§5.4 画布编译校验**(新):非法图定义编译失败→结构化错误→会话不启动。
