## 1. 执行摘要（结论先行）

**总体可行性结论：可行。** 技术栈成熟、开源生态完备、可完全自托管（含 GPU），无需依赖任何外部 SaaS。

**一句话推荐技术栈：**

> **后端 Python（FastAPI）+ LangGraph 图式编排底座**（可选叠加 Deep Agents harness）；前端 Next.js + React + TypeScript + ReactFlow；存储 PostgreSQL + pgvector + Redis；模型接入 LiteLLM 网关 + 本地 vLLM/Ollama（GPU 推理）；沙箱 **agent-infra/sandbox**（Docker 部署，GPU 透传）+ **自建 Sandbox Manager 控制面**；**协议层 MCP（工具接入）+ A2A（agent 间通信，中期）**；**知识沉淀三层（向量 RAG [LlamaIndex] + GraphRAG [LightRAG] + 结构化 Wiki）+ CRAG 检索校验**；Docker Compose（开发）→ K8s（生产）。

**核心风险提示（详见第 9 章）：**

1. **沙箱控制面需自建**——agent-infra/sandbox 只是**数据面（运行时）**，组织下**没有开源的 sandbox manager**；多实例生命周期、GPU 配额、调度全靠平台自建一层控制面，是核心工作量（详见第 6 章）。
2. **沙箱强隔离与 GPU 透传**——sandbox 默认为普通 Linux 容器（非 microVM），隔离强度弱于 E2B；GPU 透传依赖 Docker `--gpus` + NVIDIA Container Toolkit，**必须在 POC 阶段实测**。
3. **知识沉淀的工程量与质量治理**——三层知识架构（向量 RAG/GraphRAG/Wiki）是长期建设，GraphRAG 索引成本高、LLM 抽取的知识需「人工审核」保证可信（详见第 8 章）。
4. **GPU 多租户调度与本地模型准入**——团队共享 GPU 需配额/队列防独占；内网本地推理（vLLM/Ollama）的模型质量直接决定 agent 上限，需建立评测准入流程。

## 2. 项目背景与需求理解

### 2.1 核心需求拆解

| # | 需求 | 说明 |
|---|------|------|
| 1 | **BS 架构** | 浏览器访问、团队多人协作、集中部署、统一运维 |
| 2 | **模型可配置** | 多 provider 接入、参数（temperature/上下文/function calling）可调 |
| 3 | **记忆可配置** | 短期对话历史 + 长期记忆（设计案例、历史优化决策的 RAG 检索） |
| 4 | **工具可配置** | function calling + MCP 协议，挂载领域工具（仿真、优化求解器等） |
| 5 | **技能可配置** | 按需加载的领域能力包（如「气动分析」「多目标优化」） |
| 6 | **沙箱执行** | 安全隔离的代码/脚本/仿真运行环境 |
| 7 | **开源框架支持** | 复用 LangGraph 等成熟编排框架，避免重复造轮子 |
| 8 | **团队协作** | 多用户、权限、共享资产（技能/知识库/agent 模板） |

### 2.2 场景特点（无人系统设计优化研究）

- 🔮 **计算密集**：可能涉及数值仿真（CFD 气动）、优化算法（遗传/贝叶斯优化）、参数搜索——**强依赖 GPU/CPU 与沙箱执行能力**，这是「需 GPU」约束的技术根源。
- 🔮 **领域知识沉淀**：历史设计方案、优化结论、试验数据需要沉淀为长期记忆与知识库（RAG）。
- 🔮 **人机协同**：设计人员需要在 agent 执行中「人工接管 / 审核 / 修正」——要求沙箱支持可视化交互（如 VSCode Server、浏览器接管）。
- 🔮 **可追溯**：设计决策需可审计，要求 agent 执行有完整的 trace/可观测能力。

## 3. 可行性分析

### 3.1 技术可行性 —— ✅ 高

- 开源生态完备：Python agent 生态（LangGraph/LangChain/LlamaIndex）、模型网关（LiteLLM）、记忆框架（Mem0/Letta）、沙箱（agent-infra/sandbox）均成熟且 License 友好（详见附录）。
- 同类 BS 平台已验证路线：Dify（Python+Flask+Celery）、Coze Studio（Go）、FastGPT（Node.js）证明「BS agent 平台 + 可视化编排 + 沙箱」架构可行，可作为参考蓝本。
- 需求中的「模型/记忆/工具/技能」四类配置，业界已有成熟抽象（见第 4 章），无需原创设计。

### 3.2 集成可行性（agent-infra/sandbox）—— ✅ 可行

- ✅ 提供 **Python SDK 与 TypeScript SDK**（官方文档站 sandbox.agent-infra.com）。
- ✅ **Docker 自托管**，一条 `docker run` 即可起一个完整环境，内网适用。
- ✅ 通过 **MCP（Model Context Protocol）** 接入 agent，与平台「工具层」天然契合。
- 🔮 集成方式：平台编排引擎（LangGraph）在工具节点调用 Sandbox Python SDK，按 agent 会话创建/复用容器实例。

### 3.3 部署可行性（自托管 + GPU）—— ✅ 可行

- ✅ 所有组件均可自托管，无强制外部 SaaS 依赖。
- ✅ **GPU 透传成熟方案**：NVIDIA Container Toolkit + Docker `--gpus`（业界标准，vLLM/本地推理通用）。
- ✅ 内网本地推理：vLLM（高性能，GPU）、Ollama（易用，GPU/CPU 均可），均暴露 OpenAI 兼容端点。
- 🔮 生产高可用：K8s 部署，各模块独立多副本（api / worker / sandbox / model-gateway）。

### 3.4 合规可行性 —— ✅ 可商用/可内网使用

- 本方案所选核心组件均为宽松开源许可（MIT / Apache-2.0）。
- ⚠️ 已规避有附加条款的组件：**未选用** FastGPT（Apache-2.0 + 禁止 SaaS 转售条款）、**未选用** n8n（fair-code，非严格开源）。

### 4.1 后端

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | **FastAPI** | 原生 async、SSE/WebSocket 友好、自动 OpenAPI 文档；用户指定 |
| ASGI | Uvicorn / Gunicorn | 生产标配 |
| 异步任务 | **ARQ** 或 Celery | ARQ 更轻量原生 async；Celery 生态更全（Dify 用 Celery）。🔮 建议 ARQ（与 FastAPI 同为 async，更契合） |
| Agent 编排·底层 | **LangGraph** | ✅ 图式可控编排、状态/checkpoint、流式输出，MIT，生态最大；承载「模型/记忆/工具/技能」可配置编排的执行底座 |
| Agent 编排·上层 | **Deep Agents**（可选） | 🔮 LangChain 官方 harness（构建于 LangGraph 之上），原生支持子代理/规划/文件系统/skills/流式/沙箱/MCP，加速「自主 agent」搭建；详 4.1.1 |
| ORM | SQLAlchemy 2.0 + Alembic | Python 主流 |
| 鉴权 | Casbin / FastAPI 自带 | RBAC，支撑团队权限 |

#### 4.1.1 Deep Agents 与 LangGraph 的关系（关键澄清）

Deep Agents（[github.com/langchain-ai/deepagents](https://github.com/langchain-ai/deepagents)）是 **LangChain 官方**于 **2026-04 正式发布**的 agent harness，**构建于 LangGraph 之上**——它是「上层抽象」，**不是** LangGraph 的替代品。选它即自动绑定 LangGraph + LangChain 全家桶。

**为什么值得纳入考量（与需求的契合点）**：

| 需求维度 | Deep Agents 原生支持 |
|----------|---------------------|
| 子代理 / 多 agent | ✅ v0.5 起支持**后台远程 agent（非阻塞）**，多 agent 可并行而非阻塞等待 |
| 工具 / MCP | ✅ 工具调用 + MCP 原生支持 |
| 记忆 | ✅ 内置 memory |
| 规划（planning） | ✅ 内置规划能力 |
| 文件系统 | ✅ 原生 file_system 抽象（与沙箱天然契合） |
| 技能（skills） | ✅ 原生 skills 概念 |
| 沙箱 | ✅ 原生支持 |
| 流式输出 | ✅ |

> 来源：[官方博客 v0.5](https://www.langchain.com/blog/deep-agents-v0-5)｜[Towards AI 发布报道](https://pub.towardsai.net/langchain-just-released-deep-agents-and-it-changes-how-you-build-ai-systems-cc2371b04714)

**Deep Agents vs LangGraph 定位对比**：

| 维度 | LangGraph（执行底座） | Deep Agents（上层 harness） |
|------|----------------------|---------------------------|
| 抽象层级 | 图式状态机（节点/边/状态） | 「自主 agent」高层抽象（子代理/规划/文件系统/skills 预设） |
| 灵活度 | 高（自定义图拓扑） | 中（预设范式，快速上手） |
| 上手成本 | 较高 | 低（声明式定义 agent） |
| 依赖 | 独立 | 绑定 LangGraph + LangChain |

**🔮 选型建议（两种可行路线）**：

- **路线 A（推荐，稳妥）**：以 **LangGraph 为执行底座**，平台自建领域节点与编排逻辑。Deep Agents **仅借鉴其 file_system / skills / 后台子代理设计**，不引入强依赖。优点：可控、解耦、无年轻项目风险。
- **路线 B（激进，快速）**：直接用 **Deep Agents 作为 agent 运行时主体**，跑在 LangGraph 之上，快速获得「子代理/规划/文件系统/skills/沙箱/MCP」开箱能力。优点：开发快、契合度高；**风险**：项目年轻（Beta，2026-04 起）、强绑定 LangChain 生态、安全模型需信任 LLM 自主操作。

**⚠️ 风险提示**：Deep Agents 仍是 **Beta**（截至 2026-06 版本 0.6.12，发布约 2–3 个月），迭代快但 API 可能变动；其安全模型信任 LLM 自主进行文件/子代理操作，需配合沙箱强隔离（见第 5、6 章）。

#### 4.5.1 Agent 互操作协议：MCP / A2A / ACP 怎么选（关键澄清）

这三者**不在同一层**，是正交互补的两层，业界已收敛为 **MCP（工具/数据层）+ A2A（agent 间通信层）** 双层标准：

| 协议 | 层级 | 提出方 | 传输 | 核心抽象 | 当前状态（2026-06） | License |
|------|------|--------|------|----------|---------------------|---------|
| **MCP** | agent↔**工具/数据** | Anthropic | JSON-RPC | Tools/Resources/Prompts | ✅ 事实标准、生态最成熟 | MIT/Apache-2.0 |
| **A2A** | agent↔**agent** | Google Cloud（→LF） | HTTPS+JSON-RPC+SSE | Agent Card/Tasks/Artifacts | ✅ v1.0 生产级，150+ 组织采用 | Apache-2.0 |
| ~~ACP~~ | agent↔agent | IBM/BeeAI | RESTful HTTP | agent 服务发现 | ❌ **已于 2025-08 并入 A2A，已 EOL** | Apache-2.0 |

**两个决定性事实（已确认）：**
1. ✅ **ACP 已并入 A2A**（2025-08-29，Linux Foundation LF AI & Data 公告，IBM/Google 联署）——ACP 团队停止主动开发、转向贡献 A2A。"ACP vs A2A"已不是选择题，**新项目不应再基于 ACP**。
2. ✅ **MCP 与 A2A 正交互补，非竞争**——MCP 连接 agent↔工具/数据，A2A 连接 agent↔agent（Google 与 Anthropic 双方均明确此立场；典型组合：agent 经 MCP 接工具、经 A2A 与其他 agent 协作）。

**🔮 选型建议（针对本平台）：**

| 协议 | 优先级 | 理由 |
|------|--------|------|
| **MCP** | **必选**（已计划） | 工具/数据接入基座，FastAPI 无缝对接 |
| **A2A** | **中期推荐**（规划优先） | agent 间通信层唯一"活着的"开放标准；若需异构/跨团队/第三方 agent 互操作或「agent 作为服务」发现机制，是当前最佳选择 |
| **ACP** | **不采用** | 已 EOL，基于 ACP = 技术负债 |

**何时纳入 A2A？** 🔮
- **现在就该规划**：若平台 agent 需被外部系统发现/委派任务，或要接第三方/开源 agent。
- **可暂缓**：若当前是纯内部、单一 LangGraph 图编排、所有 agent 同进程同框架、无跨边界通信——先用 MCP + LangGraph 内部编排。

**若纳入 A2A，对架构的影响（🔮）：**
1. **工具协议层（MCP）不受影响**——A2A 与 MCP 正交。
2. **编排层分层**：**本地编排**走 LangGraph 原生调用（低延迟、强耦合）；**跨服务/跨边界**走 A2A（Agent Card 发现 + JSON-RPC 委派 Task）。**不必把所有 agent 通信都走 A2A**——同框架内部节点走原生调用即可。
3. **服务发现**：需一个「agent 注册/发现」目录托管 Agent Card（well-known URL，轻量增量）。
4. **鉴权**：A2A v1.0 支持 Agent Card 鉴权声明 + signed Agent Cards，对内网自托管友好，纳入统一鉴权体系。
5. FastAPI 天然契合 A2A server 端（HTTP+JSON-RPC+SSE）；A2A 官方有 Python SDK，降低接入成本。

> 来源：[LF 公告（ACP 并入 A2A）](https://lfaidata.foundation/communityblog/2025/08/29/acp-joins-forces-with-a2a-under-the-linux-foundations-lf-ai-data/)｜[Google A2A 公告](https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/)｜[A2A 仓库](https://github.com/a2aproject/A2A)｜[MCP/A2A 综述 arXiv:2505.02279](https://arxiv.org/html/2505.02279v1)

### 4.6 技能（Skills）实现思路

🔮 借鉴 Anthropic Agent Skills 概念，定义「按需加载的领域能力包」：

- 每个 Skill = prompt 指令 + 元数据 + 可选资源（脚本/模板/工具引用）。
- 运行时由 agent 按需激活（而非全塞进 system prompt，避免上下文爆炸）。
- 领域示例：「气动分析技能」「多目标优化技能」「结构轻量化技能」——技能内部可组合多个工具（仿真器 + 求解器 + 后处理）。
- 与「工作流」区别：Skill 是「按需注入的能力」，Workflow 是「预定义的确定流程」；二者正交可组合。

### 4.7 流式与实时

- **SSE**：LLM token 流式输出（主流、轻量）。
- **WebSocket**：多 agent 实时状态推送、人工接管/打断。

### 4.8 部署

- 开发：**Docker Compose**（一键起 PG+Redis+LiteLLM+vLLM+Sandbox+前后端）。
- 生产：**Kubernetes**（api/worker/sandbox/model-gateway 各自多副本 + HPA）。

## 5. 沙箱专项分析（agent-infra/sandbox）

> 一手来源：[GitHub 仓库](https://github.com/agent-infra/sandbox)｜[官方博客（中文）](https://sandbox.agent-infra.com/zh/blog/announcing-0)｜[ByteDanceOSS 介绍](https://dev.to/bytedanceoss/introducing-aio-sandbox-all-in-one-sandbox-environment-for-ai-agents-18k0)

### 5.1 项目定位

- ✅ 全称 **AIO Sandbox (All-in-One Agent Sandbox Environment)**，在**单个 Docker 容器**中集成：**Browser（浏览器）+ Terminal/Shell + File（文件系统）+ MCP 操作 + VSCode Server**。
- ✅ 解决「agent 不再需要自己拼装多个服务，开箱即用拿到完整、隔离、可定制运行环境」，同时服务人类（可视化接管 / 人工介入）——与场景的「人机协同」需求高度契合。
- ✅ 开发方：**agent-infra 团队**（ByteDance / 火山引擎 OSS 渠道发布）。🔮 agent-infra 大概率是字节内部 Agent 基础设施团队，属高置信度推测。

- ✅ 主线隔离 = **Docker 容器**（官方反复强调 single Docker container），即 **Linux 容器（namespace + cgroups）**。
- ✅ **不是** Firecracker microVM，**也不是** WebAssembly。
- ⚠️ **README 正文未把 gVisor 列为必需依赖**；「可叠加 gVisor」的说法多来自第三方文章，非一手表述。

## 6. Sandbox Manager 控制面（必读：需自建）

### 6.1 关键结论：开源体系里不存在独立 manager

✅ **经 GitHub 组织仓库列表确认**：`agent-infra` 组织下**仅有 2 个公开仓库**——`agent-infra/sandbox`（AIO Sandbox 运行时）与 `agent-infra/sandbox-sdk-go`（Go SDK）。**没有任何名为 `manager` / `controller` / `orchestrator` / `scheduler` 的独立项目**，sandbox 主仓库内也无 `manager/` / `control-plane/` 等控制面子模块。

> 一手来源：[组织主页](https://github.com/agent-infra)｜[主仓库](https://github.com/agent-infra/sandbox)｜[云部署指南](https://sandbox.agent-infra.com/guide/start/cloud-deployment)

🔮 **含义**：agent-infra/sandbox 是**数据面（data plane / 沙箱运行时）**，不是**控制面（control plane / 沙箱管理器）**。它对应 E2B/Daytona 里「被管理的那个沙箱单元」，而**不是** E2B/Daytona 本身。官方的「多实例」方案就是「在 K8s 里手写 `replicas: N` 的 Deployment」，TLS/鉴权/网络策略全靠你自己在 Ingress 层解决。

⚠️ **避坑提示**：搜索中出现的「Kubernetes Agent Sandbox CRD / GKE Agent Sandbox」是 **Google 的项目**，与 agent-infra 无关，勿混淆。

### 6.2 数据面 vs 控制面职责划分（最重要）

| 能力 | AIO Sandbox（数据面，✅ 已具备） | Sandbox Manager 控制面（🔮 需自建） |
|------|--------------------------------|--------------------------------|
| 容器内 Shell/File/Browser/MCP/VSCode 执行 | ✅ | — |
| 容器内 API 鉴权（API Key / JWT） | ✅ | — |
| 容器内 Preview Proxy / 服务监控 | ✅ | — |
| 创建/销毁 sandbox 实例 | ❌ | **必须有** |
| 实例池 / 复用 / 空闲回收 | ❌ | **必须有** |
| CPU/内存/**GPU** 配额与多租户隔离 | 仅靠外部 K8s limits | **必须有**（尤其 GPU） |
| 会话↔实例映射、亲和绑定 | ❌ | **必须有** |
| 健康检查 / 自动重启 | ❌ | 建议 |
| 日志与产物收集、审计 | ❌（容器日志） | **建议有**（设计可追溯需求） |
| 快照/暂停-恢复 | ❌ | 暂非必需，后期评估 |
| 集群/多节点调度 | ❌ | K8s 生产时需要 |

### 6.3 控制面职责定义（落入你平台的形态）

🔮 在 FastAPI 平台中自建一层 **Sandbox Manager 服务**（独立模块/微服务），承担：

1. **生命周期管理**：按 agent 会话动态拉起/销毁 `ghcr.io/agent-infra/sandbox` 容器；维护「会话→容器」映射表。
2. **实例池与复用**：预热一组空闲实例，缩短冷启动；会话结束后回收/复用。
3. **资源配额（重点=GPU）**：团队/用户级 CPU/内存/GPU 配额；GPU 采用队列+独占/共享策略，防止单会话独占。
4. **亲和与会话绑定**：同一会话尽量复用同一实例（保留中间产物/已装依赖），跨请求可恢复状态。
5. **网络与接入**：为每个实例注入 `SANDBOX_API_KEY`/JWT、生成临时 `base_url`，经平台网关代理给前端（VSCode/Browser 可视化接管）。
6. **运维**：健康检查、超时/空闲回收、日志与产物收集（关联会话供审计）。

### 6.4 技术实现选型（🔮）

| 实现方式 | 适用阶段 | 说明 |
|----------|----------|------|
| **Python Docker SDK**（`docker-py`） | **POC / 单机 MVP** | 最轻量，本机动态 `run`/`rm` 容器，`device_requests` 透传 GPU；无需 K8s |
| **Kubernetes API**（官方 `kubernetes` Python 客户端 / Job+Pod） | **生产** | 多节点调度、资源 limits、HPA、自愈；GPU 经节点 `nvidia.com/gpu` 资源声明 |
| **K8s Operator/CRD（自研）** | 后期规模化 | 声明式 `SandboxSession` CRD，reconcile 生命周期；成本较高，非首发 |

🔮 **推荐演进路径**：阶段 0–1 用 **Docker SDK**（单机，快速验证 GPU）；阶段 3 生产化切 **K8s Job/Pod**（多副本、配额、可观测）。控制面接口保持不变（统一的 `SandboxManager` 抽象层），底层从 Docker 切 K8s 时只换实现。

### 6.5 控制面伪代码骨架（概念示意）

```python
# 控制面统一抽象（底层可换 Docker / K8s 实现）
class SandboxManager:
    def acquire(self, session_id, gpu=False, cpu=2, mem="4G") -> str:
        """为会话获取/复用一个 sandbox，返回 base_url"""
        ...
    def exec(self, base_url, cmd: str) -> str:
        """委托给 AIO Sandbox 的 /v1/shell/exec"""
        ...
    def release(self, session_id, destroy=False):
        """回收或销毁实例（按配额/空闲策略）"""
        ...
    # 配额/健康检查/产物收集等 ...
```

```python
# 编排引擎（LangGraph 工具节点）经控制面用沙箱
mgr = SandboxManager(backend="docker")  # 生产时 backend="k8s"
base_url = mgr.acquire(session_id, gpu=True)
mgr.exec(base_url, "python3 optimize_uav.py")
```

### 6.6 与同类控制面对比（🔮 定位）

- **E2B / Daytona**：自带完整托管控制面（多实例编排、dashboard）。本方案用 agent-infra 时**这层要自建**——是集成的主要工作量所在。
- **Modal**：全托管 serverless 调度，不适用内网自托管。
- **自研 K8s 调度**：恰恰是你要做的事，agent-infra 只提供镜像，调度逻辑全靠你。

✅ **License 不受影响**：控制面是你自研代码，AIO Sandbox 本体 Apache-2.0 可商用。唯一变数是 Kubernetes 自身（CNCF，宽松许可）与 Docker（引擎 GPLv3 + Apache 商业组件；🔮 内网自用无合规障碍，若需完全 OSI 可换 Podman/Containerd）。

> ⚠️ **本章结论直接影响第 7 章风险与第 8 章路线图**：自建控制面是平台的核心开发工作量，建议在阶段 0–1 就把「Docker SDK 控制面 + GPU 透传」打通。

## 8. 知识沉淀能力（团队级长期资产）

> 本章对应需求「记忆可配置 + RAG / LLM Wiki / GraphRAG」。先厘清一个**关键概念边界**：平台同时存在「记忆」与「知识沉淀」两类东西，但它们的归属、时效、形态完全不同，必须分开设计，不能混为一谈。

### 8.1 核心概念：知识沉淀 ≠ 记忆（必先厘清）

| 维度 | 记忆 Memory | 知识沉淀 Knowledge Sedimentation |
|------|-------------|--------------------------------|
| 归属 | per-conversation / per-agent | **团队级、可共享** |
| 时效 | 会话级 / agent 生命周期 | 长期资产，跨会话跨人 |
| 形态 | 原始交互片段、上下文窗口 | 结构化、可检索、可审计 |
| 可合成性 | 低（原样回放） | 高（综合/归纳成 wiki/规则/图谱） |
| 谁消费 | 单个 agent 记住当前用户/任务 | **整个团队复用历史经验** |

🔮 **对本场景的判断**：无人系统设计优化研究的核心价值在于「历史案例、优化决策、领域规则、设计教训」的**团队级复用**——这正是「知识沉淀」而非「记忆」。**知识沉淀是本平台应优先、重点建设的能力**，记忆（见 4.5）是辅助。

### 8.2 推荐分层架构（三层知识资产 + 编排消费）

```
┌─────────────────────────────────────────────────────────────┐
│  Agent 编排层 (LangGraph / Deep Agents) — 消费者             │
│  · 短期记忆：会话上下文窗口（不入知识资产库）                   │
│  · Agentic RAG：把检索当工具，CRAG 式打分+改写+兜底           │
└───────────────┬─────────────────────────────────────────────┘
                │ 调用三层团队级知识资产（共享、可检索、可审计）
┌───────────────▼─────────────────────────────────────────────┐
│ 第3层  结构化知识沉淀 / Wiki 层  ← 本平台核心建设对象          │
│   设计方案/决策记录/领域规则/设计教训，LLM 抽取+人工审核       │
│   存储：PostgreSQL 关系表（带版本/时间戳/决策链）             │
├─────────────────────────────────────────────────────────────┤
│ 第2层  GraphRAG 层  ← 跨案例综合 / 实体-关系多跳推理          │
│   实现：LightRAG（MIT，支持 PostgreSQL，成本远低于 MS GraphRAG）│
│   价值：全局权衡趋势（如"所有方案在续航上的取舍规律"）         │
│   备选/重型：MS GraphRAG（需强全局摘要报告时）                │
├─────────────────────────────────────────────────────────────┤
│ 第1层  向量 RAG 层  ← 局部事实 / 文档检索（覆盖 80% 问答）     │
│   实现：LlamaIndex（代码优先、可深度定制，MIT）+ pgvector     │
│   多模态：图纸/CAD/图表 → ColQwen2 晚交互检索（需 GPU）       │
│   检索增强：query 改写 + rerank                              │
└─────────────────────────────────────────────────────────────┘
```

### 8.3 各层技术选型

| 层 | 推荐实现 | License | 存储 | 备选 |
|----|----------|---------|------|------|
| 向量 RAG | **LlamaIndex** + pgvector | MIT | PG + pgvector | Haystack（需评测时） |
| 多模态图纸 | **ColQwen2**（[illuin-tech/colpali](https://github.com/illuin-tech/colpali)） | Apache-2.0 🔮 | 多向量索引 ⚠️ | 图像 caption + 普通向量（退而求其次） |
| GraphRAG | **LightRAG**（[HKUDS/LightRAG](https://github.com/HKUDS/LightRAG)） | MIT | PG 或 Neo4j | MS GraphRAG（重型）/ HippoRAG2（多跳记忆） |
| 结构化 Wiki | **自建**（FastAPI + PG + LLM 抽取 + 人工审核工作流） | 自有 | PG 关系表 | — |
| Agentic 检索编排 | **LangGraph**（CRAG 模板） | MIT | — | Deep Agents |

### 8.4 关键技术点说明

**① GraphRAG vs 传统向量 RAG 的本质区别（何时用哪个）**

| 维度 | 传统向量 RAG（第1层） | GraphRAG（第2层） |
|------|----------------------|-------------------|
| 索引对象 | 文档块的向量 | 实体-关系图 + 社区摘要 |
| 擅长 | 局部事实、点查询（"某方案写了什么"） | 全局综合、多跳关系（"所有方案在某维度的权衡规律"） |
| 索引成本 | 低 | **高**（LLM 抽取实体+层次摘要，token 随语料激增） |

🔮 **为什么推荐 LightRAG 而非 MS GraphRAG 起步**：MS GraphRAG 全量索引 token 成本高昂（数千份案例可能消耗数百万 token），且增量更新弱、调参复杂；LightRAG（港大 HKUDS，EMNLP 2025）双层级检索、支持增量、支持 PostgreSQL，社区评测延迟与 token 消耗显著低于 MS GraphRAG 而基准持平或更优。**成本敏感 + 需增量 + 内网** → LightRAG 是更经济的选择，MS GraphRAG 作为「需强全局摘要报告」的重型备选。

**② CRAG（Corrective RAG）思想**——编排层的检索质量校验：[论文 arXiv:2401.15884](https://arxiv.org/abs/2401.15884)。引入轻量检索评估器对结果打分（correct/incorrect/ambiguous），低质时触发改写/兜底。🔮 内网环境下把 web 兜底替换为「内网二次检索 + 人工兜底」。LangGraph 官方 Self-Reflective RAG 即 CRAG 的 agent 化实现，**本场景高度推荐采用 CRAG 思想**。

**③ 多模态 RAG（图纸/CAD）**——**ColQwen2**（[ICLR 2025 ColPali](https://github.com/illuin-tech/colpali)）用 VLM 把整页文档当图像直接嵌入，晚交互多向量匹配，**无需 OCR**，高度契合无人系统图纸检索。⚠️ **需验证**：pgvector 对 late-interaction 多向量原生支持有限，可能需 Weaviate/Vespa（属第 1 层的实现变数）。

**④ 长上下文（200K+）会让 RAG 过时吗？** ✅ 业界共识：**不会**。RAG 在成本（比百万 token 填充便宜 8–82×）、数据新鲜度、噪声控制、lost-in-the-middle 衰减上仍胜出；新兴共识是「长上下文 + RAG 混合」，术语正转向「context engineering（上下文工程）」。🔮 内网案例库持续增长，长上下文无法承载全部历史 → **RAG 仍是刚需**，复杂分析可长上下文装载相关子集。

### 8.5 与平台其它能力的边界划分（关键）

| 关注点 | 归属 | 说明 |
|--------|------|------|
| 会话上下文、最近交互 | **编排层短期记忆** | 不入知识资产库 |
| 用户偏好、agent 行为经验 | **agent 长期记忆**（Letta 风格，per-agent，可选） | 非团队共享 |
| 历史案例、优化决策、领域规则、设计教训 | **知识沉淀三层**（团队级共享） | **本平台核心建设对象** |
| 图纸/CAD | 多模态 RAG 层（ColQwen2） | 需 GPU |
| 跨案例关系/权衡 | GraphRAG 层（LightRAG） | 全局综合 |

🔮 **边界原则**：记忆服务「单个 agent 记住当前用户/任务」；知识沉淀服务「整个团队复用历史经验」。编排层（LangGraph）是**消费者**，三层知识资产是**供给方**；CRAG 检索质量校验放在编排层，对三层结果统一打分/路由。

### 8.6 落地优先级建议（🔮）

1. **先建第 1 层向量 RAG**（LlamaIndex + pgvector）——成本最低、见效最快，覆盖 80% 局部问答。
2. **同步建结构化 Wiki 表**（PG 关系表 + 决策/案例 schema + 人工审核）——这是「知识沉淀」的真正主体，决定团队资产价值。
3. **再加 GraphRAG（LightRAG）**——案例积累到需跨案例综合时引入。
4. **多模态（ColQwen2）按需**——图纸检索成痛点时引入，需 GPU 预算。
5. **编排层用 LangGraph + CRAG**——把三层检索封装为工具，agent 自主路由。

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **sandbox 强隔离不足** | 多 agent 间/对宿主潜在逃逸 | 🔮 可选 gVisor(runsc) 作为容器 runtime 加强；或双层隔离；涉密前做隔离 POC |
| **sandbox GPU 透传未一手确认** | GPU 优化场景可能受阻 | 🔮 **阶段0 POC 必测**：Docker `--gpus` + NVIDIA Container Toolkit 跑 GPU 脚本；不达标则切 Daytona |
| **控制面需自建（核心工作量）** | 生命周期/配额/调度全靠平台实现 | 🔮 自建 Sandbox Manager（Docker SDK→K8s 演进），统一抽象层；详见第 6 章 |
| **Deep Agents 引入决策** | 选其则强绑定 LangChain 生态、项目年轻(Beta) | 🔮 路线 A（仅借鉴设计）稳妥；路线 B（作运行时主体）需接受 Beta 风险；详见 4.1.1 |
| **GPU 多租户调度** | 单会话独占、资源争抢 | 🔮 控制面加队列+配额+独占/共享策略；监控显存占用 |
| **本地模型能力与合规** | 内网模型质量决定 agent 上限 | 🔮 vLLM/Ollama + 建立「模型评测准入流程」（领域任务基准） |
| **LangGraph 学习成本** | 团队上手曲线 | 🔮 封装领域节点（无人系统优化专用），屏蔽底层复杂度 |
| **快照/恢复缺失** | 无法快速克隆 agent 运行态 | 🔮 先不依赖；若强需则评估 E2B/Daytona 补位 |
| **GraphRAG 索引成本高** | MS GraphRAG 全量索引 token 消耗大 | 🔮 起步用 LightRAG（成本低、支持增量）；MS GraphRAG 仅在需强全局摘要时启用 |
| **多模态多向量检索兼容性** | ColQwen2 晚交互需专用向量索引 | ⚠️ pgvector 对 late-interaction 多向量原生支持有限，需 POC 验证，必要时引入 Weaviate/Vespa |
| **知识质量与幻觉** | LLM 抽取/归纳的知识可能不准 | 🔮 结构化 Wiki 层设「LLM 抽取 + 人工审核」工作流，保证团队资产可信可审计 |
| **License 风险** | 商用/合规 | ✅ 已规避有附加条款组件（FastGPT/n8n 未选用）；控制面为自研代码，详见附录 |

## 10. 实施路线图（分阶段）

| 阶段 | 目标 | 关键交付 |
|------|------|----------|
| **阶段 0 · POC** | 打通核心硬骨头 | 沙箱 + GPU 透传 + 本地模型（vLLM）跑通一条「优化脚本」链路；**自建 Sandbox Manager 雏形（Docker SDK 控制面）**；**向量 RAG（LlamaIndex+pgvector）最小验证**；验证 sandbox Python SDK 可控 |
| **阶段 1 · MVP** | 单 agent 可用 | 四类配置面板 + 单 agent LangGraph 编排 + SSE 流式 + 沙箱执行；**第1层向量 RAG + 第3层结构化 Wiki 表（含人工审核流）**；控制面：会话→容器映射 + 空闲回收；基础鉴权 |
| **阶段 2 · 协作与编排** | 团队价值 | ReactFlow 工作流画布 + 多 agent + 技能市场；团队/权限/共享知识资产；**第2层 GraphRAG（LightRAG）+ CRAG 检索校验**；按需引入多模态（ColQwen2）；控制面：GPU 配额与产物收集 |
| **阶段 3 · 生产化** | 可运维 | 控制面切 K8s Job/Pod + 多节点调度 + 可观测（trace/metrics）+ 权限/审计 + 备份；**知识资产治理（版本/溯源/增量更新）** |

🔮 建议 **阶段 0 不超过 2–3 周**，重点验证 sandbox GPU 这一首要风险点，再投入大规模开发。
